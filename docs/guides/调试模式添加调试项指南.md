## 概述

---

调试模式是面向开发者的诊断能力：通过隐藏入口进入，勾选调试项后，主界面会显示与所选调试项相关的按钮、状态行等。本文前半部分说明**如何进入与退出**；后半部分说明**如何新增调试项**。

请严格遵循 [CONTRIBUTING.md](../../CONTRIBUTING.md) 进行代码与文档编写。

## 使用说明（进入 / 退出 / 界面）

---

### 什么是调试模式？

启用后，应用根据已勾选的**调试项**展示额外 UI（例如 TTS 信息导出按钮、状态区「当前启用的调试项」等），用于验证功能或输出诊断数据。未勾选的调试项不会生效。

### 如何进入？

| 步骤 | 操作 |
|------|------|
| 1 | 启动应用，进入主界面。 |
| 2 | 在 Toolbar **连续点击应用名称区域（「语道」）共 7 次**。每次点击标题会有**语流轻舒、道字落点**反馈（字间距如出声轻舒，「道」字短暂提亮放大；`DebugModeUi.playTitleTapFeedback`），并伴随短触觉反馈。点击间隔超过 **2 秒** 时计数清零。 |
| 3 | 从第 **4** 次点击起，Toast 会即时提示「再点击 N 次打开调试面板」（使用 `ToastHelper.showOrUpdateShort`，连点时只更新文案、不闪烁）。 |
| 4 | 第 7 次点击后弹出 **调试面板**；勾选需要的项，点击 **启用**。 |
| 5 | 主界面标题变为「语道（调试模式）」；**退出调试模式** 位于右上角 **⋮** 溢出菜单中（不占 Toolbar 条宽度，避免长标题被挤压）。 |

**说明**：调试模式入口**不在**右上角「⋮」菜单中；仅通过连点标题进入。调试模式开启后，仍可再连点标题 7 次打开同一面板，修改已选调试项。

### 如何退出？

| 方式 | 说明 |
|------|------|
| **⋮** 菜单中的 **退出调试模式** | 立即清空所有调试项并恢复普通界面。 |
| 调试面板中不勾选任何项后点 **启用** | 等同退出；Toast 提示「已退出调试模式」。 |
| 调试面板点 **取消** | 关闭面板，**不改变**当前是否处于调试模式。 |
| **双击返回退出应用** | 调用 `TestManager.resetAll()` 后 `finishAffinity()`，下次再打开不会保留调试模式。旋转屏幕、切换界面语言等配置变更重建**不会**走此路径，调试模式**保留**。仅切到后台（未双击退出）时，调试模式仍保持。 |

### 界面与配色

与调试模式相关的可见元素使用 **`@color/debug_mode_accent`**（深橙 `#FF5722`），由 `DebugModeUi` 集中着色。包括但不限于：

- Toolbar 标题后缀「（调试模式）」
- **⋮** 溢出菜单中的 **退出调试模式**（`DebugModeUi.applyExitDebugModeMenuItemStyle`）
- 主界面调试按钮（如 **设置 TTS 信息保存目录**、**输出 TTS 信息**）
- 状态信息区块下部 **调试信息** 分区（`debug_mode_accent` 小标题与项标签）中的 **TTS 信息保存目录**、**当前 Voice**、**当前启用的调试项**
- 调试面板底部的 **启用** 按钮（实心橙）

**调试面板**采用分层配色，与主界面区分：对话框背后 **45% 遮罩**；面板为暖色浅底（`debug_mode_panel_bg`）+ **2dp 橙色描边**；顶栏浅橙底 + 橙色标题；列表置于白色内容槽内，项选中时橙框/浅橙底；底部 **取消** / **启用** 固定在面板底栏（列表区域单独滚动，面板最大高度约为屏幕高度的 85%）。列表正文与说明仍以深灰/浅灰保证可读。新增调试相关控件时，主界面调试元素复用 `debug_mode_accent` 或 `DebugModeUi`。

### 说明弹窗排版

所有带 HTML 的说明文案（主界面 ℹ️、编辑器、TTS 浏览器、调试面板等）经 `DialogHelper.showInfoDialog` / `InfoIconHelper` 统一走 `HtmlBulletHelper.formatInfoMessage`：列表项以 `•` 开头时自动悬挂缩进；说明文案不使用表情符号，标题用 `<b>` 分段即可。

### 代码中的主要位置

| 职责 | 路径 |
|------|------|
| 连点入口、Toolbar、状态区、调试按钮 | `app/.../ui/MainActivity.java`（`// region 调试模式`） |
| 调试项数据与开关 | `app/.../ui/test/TestManager.java`、`TestCase.java` |
| 调试面板对话框 | `app/.../ui/test/TestModeDialog.java`、`res/layout/test_mode_dialog.xml`（标题文案 `test_mode_dialog_title`：调试面板） |
| 统一配色与标题连点反馈 | `app/.../ui/test/DebugModeUi.java`（`playTitleTapFeedback` 等）、`res/values/colors.xml`（`debug_mode_accent`） |
| 退出菜单项 | `res/menu/menu_main.xml` → `action_exit_debug_mode`（`showAsAction="never"`，仅出现在 ⋮ 菜单） |

类名仍含 `Test` 前缀（如 `TestManager`、`TestModeDialog`），用户可见文案为「调试模式 / 调试项」。

### 什么是调试项？

调试项是调试模式下可单独启用的功能点，每个调试项有唯一 `id`（如 `log_tts_voices`），可多选。勾选后才会显示或执行该调试项对应的 UI 与逻辑。

## 基本原理

---

调试项由 `TestCase` 类表示，所有调试项统一由 `TestManager` 管理，并在主界面初始化和显示。每个调试项包含唯一的 `id`、显示名称、描述等信息。

## 添加流程

---

### 步骤 1：在 TestManager 中注册调试项

打开 `app/src/main/java/com/citadawn/speechapp/ui/test/TestManager.java`，在构造方法中添加新的 `TestCase` 实例：

```java
// TestManager.java
// region 调试项注册
testCases.add(new TestCase("your_test_id", "", ""));
// endregion
```

**参数说明**：

- `your_test_id`：调试项唯一标识（建议用英文小写+下划线）
- 名称和描述可先留空，稍后在主界面初始化

### 步骤 2：在主界面初始化调试项名称和描述

打开 `MainActivity.java`，找到 `initializeTestCases()` 方法，按如下方式为新调试项设置名称和描述：

```java
// MainActivity.java
// region 调试模式
private void initializeTestCases() {
    for (TestCase tc : testCases) {
        if ("your_test_id".equals(tc.id)) {
            tc.name = getString(R.string.your_test_name);
            tc.description = getString(R.string.your_test_desc);
            break;
        }
    }
}
// endregion
```

**说明**：

- `your_test_name` 和 `your_test_desc` 为字符串资源 ID，需在下一步添加
- 相关逻辑放在 `// region 调试模式 // endregion` 分区中

### 步骤 3：添加多语言字符串资源

在 `app/src/main/res/values/strings.xml` 和 `app/src/main/res/values-zh/strings.xml` 中分别添加英文和中文描述：

```xml
<!-- values/strings.xml -->
<string name="your_test_name">Your Debug Item Name</string>
<string name="your_test_desc">Description for your debug item.</string>

<!-- values-zh/strings.xml -->
<string name="your_test_name">你的调试项名称</string>
<string name="your_test_desc">该调试项的中文描述。</string>
```

**规范要求**：

- 中英资源文件保持相同的 `name` 属性
- 字符串资源 ID 使用小写+下划线命名
- 描述内容应清晰说明调试项的功能和用途

### 步骤 4：在功能代码中使用新调试项

在需要响应该调试项的功能代码中，通过 `TestManager.getInstance().getSelectedTestCases()` 判断是否选中：

```java
// 在相关功能代码中添加调试逻辑
for (TestCase tc : TestManager.getInstance().getSelectedTestCases()) {
    if ("your_test_id".equals(tc.id)) {
        // 执行你的调试逻辑
        Log.d("MainActivity", "执行调试项: " + tc.name);

        // 示例：模拟异常场景
        // throw new RuntimeException("模拟调试异常");

        // 示例：输出诊断信息
        // Log.i("MainActivity", "当前语速设置: " + currentSpeed);

        break; // 找到后退出循环
    }
}
```

### 步骤 5：若调试项需要额外 UI

1. **可见性**：在 `updateStatusInfo()`、`updateTtsInfoDirButtonVisibility()` 等现有方法中，根据 `TestManager.getInstance().isTestMode()` 与 `getSelectedTestCases()` 控制显示。
2. **配色**：按钮调用 `DebugModeUi.styleDebugButton()`；标签与值调用 `DebugModeUi.applyAccentText()`。路径与 Voice 等长文本在值行右侧提供复制按钮（`ic_copy`），点击写入剪贴板并 Toast 提示。
3. **布局**：状态信息类调试项放在 `activity_main.xml` 的 `layoutStatusDebugSection` 内；调试专用按钮背景使用 `@drawable/btn_debug_mode_bg`，文字使用 `@color/debug_mode_accent` 或白字（实心橙底按钮）。
4. **字符串**：按钮与状态标签文案写入 `values` / `values-zh`，并在调试项 `description` 中说明入口与触发方式。

## 完整示例

---

### 示例：添加「模拟网络异常」调试项

#### 1. 在 TestManager 中注册

```java
// TestManager.java
// region 调试项注册
testCases.add(new TestCase("simulate_network_error", "", ""));
// endregion
```

#### 2. 在主界面初始化

```java
// MainActivity.java
// region 调试模式
private void initializeTestCases() {
    for (TestCase tc : testCases) {
        if ("simulate_network_error".equals(tc.id)) {
            tc.name = getString(R.string.test_simulate_network_error);
            tc.description = getString(R.string.test_simulate_network_error_desc);
            break;
        }
    }
}
// endregion
```

#### 3. 添加字符串资源

```xml
<!-- values/strings.xml -->
<string name="test_simulate_network_error">Simulate Network Error</string>
<string name="test_simulate_network_error_desc">Simulate network connection failure for debugging error handling.</string>

<!-- values-zh/strings.xml -->
<string name="test_simulate_network_error">模拟网络异常</string>
<string name="test_simulate_network_error_desc">模拟网络连接失败，用于调试错误处理机制。</string>
```

#### 4. 在功能代码中使用

```java
// 在网络请求相关代码中添加
for (TestCase tc : TestManager.getInstance().getSelectedTestCases()) {
    if ("simulate_network_error".equals(tc.id)) {
        Log.d("MainActivity", "模拟网络异常调试");
        throw new IOException("模拟网络连接失败");
    }
}
```

### 内置示例：输出 TTS 引擎信息（`log_tts_voices`）

该调试项通过主界面 **输出 TTS 引擎信息** 一次导出两个文件（与朗读按钮无关）。

- **保存目录**：在状态信息区块 **TTS 信息保存目录** 中展示；已设置自定义目录时显示所选路径，未设置时显示应用外部存储默认目录。路径右侧可复制。
- **`tts_info_[引擎名称].txt`**：语言与发音人（Voice）可读清单；标注引擎默认语言、默认发音人及各语言的默认发音人。
- **`tts_locales_[引擎名称].txt`**：`getAvailableLanguages()` 每个 `Locale` 的方法字段（`属性 (方法): 值`），供 Locale 对象诊断；与 `tts_info` 分文件存放，避免单文件过长难检索。`display*` 使用 `LocaleHelper.getCurrentLocale()`。
- **保存反馈**：两个文件均成功时 Toast 提示已保存两个文件；完整目录路径以状态信息为准。

### 内置示例：显示当前 Voice（`show_current_voice`）

启用后，在状态信息区块 **当前 Voice** 中实时展示 `TextToSpeech.getVoice()` 返回的 `Voice`（`toString()`；为 null 时单独标注）。TTS 就绪且 `getVoice()` 非 null 时显示复制按钮。不取下拉框选中项。不写入 Logcat。

### 内置示例：一键填入测试文本（`fill_test_text`）

启用后，主界面文本框下方显示 **一键填入测试文本** 按钮。点击后用内置样例（`values` / `values-zh` 中的 `debug_test_text_sample`，分别为英文与中文长文）替换当前输入，并 Toast 提示已填入；样例长度超过 TTS 引擎上限时自动截断。

## 注意事项

---

### 必须遵循的规范

- **调试项 ID**：必须唯一，且不可变，建议使用英文小写+下划线命名
- **国际化**：名称和描述必须支持中英文国际化
- **资源管理**：
    - **字符串资源**：所有文本内容必须使用 `@string/*` 资源引用，禁止硬编码（无论使用频率如何）
    - **颜色资源**：高频使用（3次以上）使用资源或主题属性，低频使用（1-2次）可以硬编码
    - **尺寸资源**：高频使用（3次以上）使用 `@dimen` 资源，低频使用（1-2次）可以硬编码
- **代码分区**：在涉及 UI 的类中，相关逻辑放在 `// region 调试模式 // endregion` 分区中

### 开发建议

- **显示顺序**：调试项的显示顺序与添加到 `TestManager` 的顺序一致
- **性能考虑**：不要在 UI 线程中执行耗时调试逻辑
- **错误处理**：调试逻辑应包含适当的异常处理，避免影响应用稳定性
- **资源清理**：调试完成后应及时清理相关资源

### 常见问题

1. **找不到调试入口**：连点的是 Toolbar 上的 **「语道」标题**，不是菜单；需在 2 秒内连续点满 7 次。
2. **连点无 Toast**：第 4～6 次应有「再点击 N 次…」提示；若仍无，确认未在旧进程上测试（可完全退出应用后重试）。
3. **双击退出后再打开调试模式消失**：属预期行为，退出前会 `resetAll()`；调试状态不持久化到磁盘。旋转屏幕或切换界面语言不会清空调试模式。
4. **调试项不显示**：检查是否在 `TestManager` 中注册、字符串资源是否齐全、是否在 `initializeTestCases()` 中初始化名称与描述。
5. **调试逻辑不执行**：确认调试项 `id` 与注册一致，且已启用调试模式并在面板中勾选该项。
6. **国际化不生效**：检查中英文字符串资源的 `name` 是否一致。
7. **应用崩溃**：调试逻辑应加异常处理，避免影响主流程；耗时操作勿放在 UI 线程。
